通用结构化问题分析与解决框架。从理解问题到验证结果，走完完整闭环。

## 小白使用指南 🐣

别怕，这玩意儿看着复杂，用起来就三步：

### 第一步：把它丢给你的 AI 助手

直接复制 [SKILL.md](./SKILL.md) 的全部内容，粘贴到 ChatGPT / Claude / DeepSeek / Hermes 的对话里，告诉它：

> "请按照这份 Problem Solver / 事儿通 框架来处理我的问题。"

就这一句，AI 会自动按流程干活。

### 第二步：说人话，描述你的问题

不用学任何格式，像跟朋友吐槽一样说就行：

- "我的网站突然打不开了，救命" → AI 会自动识别为 🔴 紧急事故，先给诊断命令
- "我想学 Python，但不知道从哪儿开始" → AI 会识别为 🟢 学习规划，走完整引导
- "我们项目不知道该用 MySQL 还是 PostgreSQL" → AI 会帮你对比选型

你甚至可以这样说："我也不知道这算不算问题，就是感觉哪里不对劲……"

### 第三步：跟着 AI 的节奏走

AI 会在关键节点停下来（你会看到 `[BREAK — WAITING FOR YOUR RESPONSE]`），这时候：

- 它问你"我理解得对吗" → 你说是或不是，不是就纠正它
- 它给你诊断命令 → 复制粘贴到终端跑一下，把结果贴回给 AI
- 它列出方案让你审批 → 你觉得行就说"可以"，不行就说哪里不行
- 它问你要不要直接帮你改 → 你可以说"先让我看看，别急着动手"

**核心心法：你是老板，AI 是你的技术顾问。它出方案，你拍板。**

### 常见场景速查

| 你遇到的事 | 怎么用 |
|-----------|--------|
| 代码报错了 | 把报错信息完整贴出来，说"帮我分析" |
| 不知道选哪个技术 | 说"帮我对比 A 和 B，我项目情况是……" |
| 想学新东西但没方向 | 说"我是 XX 背景，想学 YY，给我规划路线" |
| 系统/服务出故障了 | 说症状 + 什么时候开始的 + 有什么异常 |
| 跟同事意见不合纠结 | 说"我的看法是 A，对方说 B，帮我客观分析" |

### 记住三个原则

1. **信息给得越完整，AI 判断越准** — 日志、截图、报错全文都往上贴，别怕啰嗦
2. **AI 不动你一分钱的东西，除非你点头** — 它只会读不会改，除非你明确说"帮我改"
3. **说人话就行** — 不用学框架术语，那是 AI 的事，你只管描述问题

## 核心理念

- 先理解再行动 — 搞清问题本质，而非急于求解
- 结构化但不僵化 — 根据复杂度和紧急度灵活调整
- 迭代优于完美 — 先交付可用方案，再持续精进
- 知己所不知 — 能力边界之外，果断承认无法解决
- 重要结论需要证据 — 诊断、安全建议、版本相关声明必须有出处

## 完整流程

```
Precheck → P0a 问题框架 → P0b 约束与目标 → P1 信息收集 → P2 方案架构
                                                    ↓
                                              P3 执行蓝图 → P3.5 执行 → P4 验证 → P5 评估
```

## 风险等级体系 (R0–R3)

| 等级 | 标签 | 说明 |
|------|------|------|
| R0 | 只读 | 查看文件、检查日志、搜索代码 — Agent 可自由执行 |
| R1 | 本地 | 本地测试、不接触生产环境/外部服务/数据库 — 确认框架后执行 |
| R2 | 状态变更可逆 | 修改文件、更新配置、安装包、重启服务 — 需每步用户明确批准 |
| R3 | 高风险 | 删除数据、修改生产系统、安全/支付配置变更 — 需完整可见性+回滚方案 |

## 场景适配

| 场景 | 流程 |
|------|------|
| 🔴 紧急事故 | Immediate Triage → Safe R0 诊断 → P3 → P3.5 → P4 |
| 🟡 技术选型 | P0a→P0b→P1→P2→P5 |
| 🟡 Bug 调试 | P0a→P0b→P1(快速)→P3→P3.5→P4 |
| 🟢 系统设计 / 学习规划 | 完整流程 |
| 低风险本地调试 | P0-lite → 本地证据 → 假设 → R0/R1 检查 → 修复建议 → P4 |

## 关键安全规则

- P0a/P0b/P2 后必须 BREAK 等待用户确认
- P2 批准 ≠ R2/R3 执行授权
- 用户未回复 ≠ 同意
- 只读诊断可不等确认直接提供（紧急场景）
